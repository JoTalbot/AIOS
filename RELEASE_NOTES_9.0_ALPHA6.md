# AIOS 9.0.0-alpha.6 — DevicePool, YAML-каталог платформ, MCP-профили

**Дата:** 2026-07-21 · **Тесты:** 693/693 зелёные · Python ≥ 3.10

Седьмой выпуск развивает мультиплатформенную архитектуру alpha.5:
теперь аккаунты получают выделенные устройства из пула, платформы
регистрируются из файлов, а MCP-инструменты работают от имени любого
профиля.

## Главное

### DevicePool — пул устройств с арендой под профили
- Принцип **one-device-one-profile**: устройство обслуживает ровно один
  аккаунт в каждый момент — защита от перекрёстных сессий.
- `lease()` идемпотентна (повторная аренда профиля продлевает его же
  устройство), выбирает наименее недавно использованный idle, умеет
  привязку к конкретному serial и синхронизацию `device_serial` в реестр
  профилей.
- `heartbeat()` + `reap_stale()`: молчащие устройства → offline, их
  аренды освобождаются автоматически.
- **CLI**: `aios devices register|list|lease|release|heartbeat|reap`.
- **REST**: `GET /api/v1/devices`, `POST /api/v1/devices/register|lease|
  release|heartbeat|reap`.
- Персистентность: `data/devices.sqlite` (`AIOS_DEVICES_DB`).

### YAML-каталог платформ — реестр как данные
- `load_catalog("platforms/")` регистрирует платформы из `*.yaml`:
  код не нужен, обновление конфигурации без рестарта.
- Эталон: `platforms/olx.yaml`. Спека: `name`, `android_package`,
  `storage_class`/`adb_class` (dotted-пути), локаль, описание,
  legacy-БД.
- Шаг к 10000+ приложений: дальше — кодогенерация CLI/REST из
  дескрипторов (см. обновлённый `docs/PLATFORMS_SCALING.md`).

### MCP: профили в инструментах
- `olx_market_stats`, `olx_listing_recommend`, `olx_price_drops`
  принимают необязательный `profile` — хранилище резолвится из реестра
  профилей (`AIOS_PROFILES_DB`) и кэшируется по имени.

### Персистентный реестр профилей по умолчанию
- `ProfileStore.default()` без env теперь использует
  `data/profiles.sqlite` — профили, созданные одной CLI-командой, видны
  всем последующим вызовам.

## Цифры
- 693 автотеста (+26), ~46 REST-роутов, 33+ CLI-команды.
- Философия guarded-action неизменна: пул и профили определяют лишь то,
  *кто* и *на каком устройстве* выполняет уже существующие операции.
